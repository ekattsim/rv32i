* Project Description

This repository contains a "Bare metal" RISC-V execution environment implementation. This implementation defines the following execution environment interface (EEI):

- A single unprivileged hart with initial state defined by power-on reset.

- The base integer ISA RV32I with FENCE, ECALL, and EBREAK executed as NOPs.

- A subset of the Zicsr and Zicntr extensions, namely the instructions:
  - CSRRW
  - CSRRS
  - RDCYCLE

- Available CSRs:
  - MTVEC
  - MEPC
  - MCAUSE
  - CYCLE

- Supported Exceptions:
  - Instruction address misaligned
  - Illegal Instruction
  - Load address misaligned
  - Store address misaligned

* Design and Implementation

[[file:hw/Core.vhd][Core.vhd]] implements a standard five-stage in-order pipeline: fetch, decode, execute, memory, and writeback. The design stays close to the textbook RV32I structure while handling the practical issues that arise in a real implementation.

** Architecture Overview

The datapath diagram below encodes the essentials of the design while keeping it easy to iterate on. The tradeoff is that it does not serve as a strict implementation specification.

[[file:resources/rv32i-datapath.png]]

One deliberate departure from the textbook pipeline is that control transfers are resolved in decode rather than execute. Conditional branches, JAL, and JALR are all decided in the ID stage, which reduces the taken-branch penalty at the cost of decode-stage forwarding and more careful stall logic.

** Hazards and Forwarding

The core uses bypassing at two points in the pipeline. Execute-stage forwarding selects operands from EX/MEM or MEM/WB before they enter the ALU, avoiding stalls for back-to-back arithmetic dependencies. Decode-stage forwarding feeds branch, JALR, and CSR consumers from EX/MEM when the needed value was just produced by the ALU.

Not every dependency can be bypassed. The hazard unit stalls fetch and decode when:

- Load-Use: a load result is consumed by the immediately following instruction
- Decode-Consumer: decode is trying to resolve a branch, ~JALR~, or CSR access before the required producer has made its value available
  
On a stall, the PC and IF/ID registers are held while ID/EX is turned into a bubble.

** Memory Interface

The core assumes tightly-coupled memory (TCM) and exposes a simple interface:

- ~memEn~ marks a load or store in the MEM stage
- ~writeEn~ distinguishes stores from loads
- ~byteEn~ selects byte, halfword, or word accesses
- ~dataAddr~ is the effective address from the ALU
- ~dataWrite~ carries the forwarded store data
  
Loads are sign-extended or zero-extended in the MEM stage based on the decoded load variant. This interface matches the cocotb memory model used by the repository and keeps the core decoupled from any particular bus fabric.

** CSR and Exception Handling

The implemented CSR subset is small:

- ~mtvec~
- ~mepc~
- ~mcause~
- ~cycle~
  
CSR reads are asynchronous in decode, while CSR writes commit on the /rising edge/ unlike the register file. The ~cycle~ CSR increments every cycle. ~CSRRW~ and ~CSRRS~ are implemented, which is enough to support the documented ~Zicsr/Zicntr~ subset, including ~RDCYCLE~.

Exception handling is minimal. The core detects:

- illegal instructions in decode
- instruction-address misalignment on taken branches and jumps
- load-address misalignment in execute
- store-address misalignment in execute
  
On an exception, the core captures the faulting PC into ~mepc~, writes the cause code into ~mcause~, flushes the later instructions in the pipeline, and redirects the next fetch to ~mtvec~.

* Building and Running Programs

The repository includes a standard bare-metal software + cocotb simulation flow for running arbitrary RV32I programs.

** Prerequisites

- RISC-V GNU toolchain (must provide ~riscv64-unknown-elf-gcc~)
- GHDL
- Python dependencies via ~uv~

** Quick Start

Run the [[file:sw/examples/test.c][example program:]]

#+begin_src sh
uv run --python 3.13 --with cocotb --with pyelftools make sim
#+end_src

Run your own program:

#+begin_src sh
uv run --python 3.13 --with cocotb --with pyelftools make sim PROG=path/to/program.c
#+end_src

** Execution Environment and Configuration

*** Pass/Fail and Host I/O

[[file:sw/crt0.S]] signals program termination to the host through MMIO:

- ~TOHOST_ADDR~ (default ~0x10000000~): write 32-bit exit code
  - ~0~ => PASS
  - non-zero => FAIL (value is reported)
    
The program can also produce console output by writing bytes to ~UART_TX_ADDR~ (default ~0x10000004~).

The cocotb testbench can also persist the full UART transcript if ~UART_LOG_PATH~ is set in the environment before invoking the simulator. This is consumed by [[file:tb/test_core.py]] directly, not via a ~make~ variable.

Example:

#+begin_src sh
uv run --python 3.13 --with cocotb --with pyelftools \
  env UART_LOG_PATH=/tmp/program.uart.log make sim
#+end_src

*** Runtime Internals (what runs during ~make sim~)

1. Build program ELF with ~riscv64-unknown-elf-gcc~ using:
   - startup: [[file:sw/crt0.S]]
   - linker script: [[file:sw/linker.ld]]
2. Launch GHDL on [[file:hw/Core.vhd]].
3. Cocotb ([[file:tb/test_core.py]]) loads ELF segments into the memory model ([[file:tb/memory_model.py]]).
4. Simulation runs until a tohost write or timeout.

*** Configuration Variables

All knobs are set via ~make~ variables:

- Program/build:
  - ~PROG~ (default ~sw/examples/test.c~)
  - ~MEM_SIZE~ (default ~64000~ bytes)
  - ~RAM_BASE~ (default ~0x00000000~)
  - ~TEXT_BASE~ (default ~$(RAM_BASE)~)
- Host/MMIO:
  - ~TOHOST_ADDR~ (default ~0x10000000~)
  - ~UART_TX_ADDR~ (default ~0x10000004~)
- Simulation:
  - ~MAX_CYCLES~ (default ~200000~)
  - ~SIM~ (default ~ghdl~)

Example:

#+begin_src sh
uv run --python 3.13 --with cocotb --with pyelftools \
  make sim PROG=sw/examples/test.c MEM_SIZE=131072 MAX_CYCLES=500000
#+end_src

* RISC-V Architectural Certification Tests

This repository uses the upstream ACT4 flow from [[https://github.com/riscv/riscv-arch-test][riscv-arch-test]] to verify the correctness of the RV32I implementation. The bundled ACT configuration under ~archtest/config/rv32i-core~ describes this design's execution environment and is used to generate self-checking architectural tests for the core.

The verification flow has two steps:

1. ACT4 generates self-checking ELFs for this DUT configuration.
2. The repo runs each generated ELF on the cocotb/GHDL simulation flow and records a summary of pass/fail results.

** Prerequisites

In addition to the base simulation flow, we need these on the ~PATH~:

- ~sail_riscv_sim~ ([[https://github.com/riscv/sail-riscv/releases/tag/0.10][0.10]] release)
- ~podman~
- ~riscv64-unknown-elf-objdump~

Also initialize the ~riscv-arch-test~ submodule.

#+begin_src sh
git submodule update --init --recursive
#+end_src

** Verification

Generate the ACT4 ELFs for this core:

#+begin_src sh
make archtest-gen
#+end_src

This invokes ACT4 with the repo's bundled DUT configuration and writes the generated ELFs under ~archtest/work/rv32i-core/elfs~.

Run those ELFs on the simulated core:

#+begin_src sh
make archtest-run
#+end_src

This runs every generated ELF through the cocotb/GHDL DUT flow and writes a summary report to ~archtest/work/reports/test_summary.json~. The runner also saves per-test stdout, stderr, and UART logs under ~archtest/work/reports/logs/~.

You can also run the full verification flow end to end:

#+begin_src sh
make archtest
#+end_src

** Interpreting Results

A successful run means the generated ACT subset passed on this RV32I core. The summary JSON records the total number of executed tests, how many passed, and the log paths for each test case.

If you made any changes and a run fails:

- inspect ~archtest/work/reports/test_summary.json~ for the failing test entry
- inspect the corresponding files under ~archtest/work/reports/logs/~
- use the saved UART transcript to correlate the ACT failure with the DUT behavior seen by [[file:tb/test_core.py]]

* Dhrystone Benchmark

This repository uses SiFive's [[https://github.com/sifive/benchmark-dhrystone][Dhrystone Benchmark]] to measure the CPI performance of the RV32I core. The benchmark sources are included as a git submodule under ~external/benchmark-dhrystone~ and are built without modifying upstream code. The runtime support layer is under ~sw/dhrystone~, which includes a timing shim that reports cycle counts using the core's ~rdcycle~ counter.

The reported performance number for this core at the ~-O2~ optimization level is *1.023 DMIPS/MHz*. This is derived from the benchmark's cycles per iteration like this:

$$\frac{\text{DMIPS}}{\text{MHz}}= \frac{10^6}{1757\cdot \left( \text{cycles per iteration} \right)}$$

** Reproducing The Benchmark

Initialize the Dhrystone submodule:

#+begin_src sh
git submodule update --init --recursive
#+end_src

Run the benchmark end to end with:

#+begin_src sh
make dhrystone
#+end_src

This flow:

1. Builds ~external/benchmark-dhrystone~ into ~build/dhrystone.elf~.
2. Runs that ELF on the RV32I core through the existing cocotb/GHDL simulation flow.
3. Saves the UART transcript to ~build/dhrystone.uart.log~.
4. Extracts the total measured benchmark cycles from the UART transcript.
5. Reports cycles per iteration and ~DMIPS/MHz~.
